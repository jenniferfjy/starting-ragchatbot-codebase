Codebase Overview

Course Materials RAG System

A full-stack Retrieval-Augmented Generation app that answers questions about course materials using ChromaDB for vector search and Anthropic Claude for response generation.

Python 3.13 FastAPI Claude (Anthropic) ChromaDB sentence-transformers Vanilla JS uv

What this app does

In one sentence: You ask a question about a course → the app finds the most relevant passages from the course material → Claude writes a sourced answer.

RAG = Retrieval-Augmented Generation. Instead of asking the AI to answer purely from its training memory, we first retrieve real text from a database, then ask the AI to answer using that text — much more accurate and verifiable.

Request flow diagram

Each column is a real piece of the system (a "lifeline"). The numbered arrows show exactly what happens, in order, when you ask a question. Solid orange arrows are forward requests; dashed grey arrows are responses returning back.

What each phase is doing

StepsPhaseWhat's happening
1–2CaptureUser types in the chat box; browser POSTs the question.
3–4Memory lookupBackend pulls the last 2 exchanges from the in-memory session store to give Claude context.
5–6First Claude callClaude sees the question + history + an available search_course_content tool — and decides to search.
7–8Database searchSearch tool resolves a possibly-partial course name against the course_catalog, then queries course_content for the top 5 most relevant chunks.
9–10Second Claude callTool results sent back to Claude (no tools this time, so it must answer). Claude writes the final response.
11Memory saveThe new Q+A is appended to session history for next time.
12–13RenderJSON returned to the browser; the chat UI displays the answer with its sources.
💡 Why two Claude calls? Claude needs to see the search results before it can write an answer. The first call decides "I need to search"; the second call writes the actual answer using what was found. This is how Anthropic's tool use works.

app.py — backend/app.py

  • FastAPI entrypoint. Title: "Course Materials RAG System".
  • Adds TrustedHostMiddleware (allowed_hosts=["*"]) and CORS (allow_origins=["*"]) for proxy compatibility.
  • Instantiates a single global RAGSystem(config).
  • Exposes two endpoints: POST /api/query and GET /api/courses.
  • @app.on_event("startup"): calls rag_system.add_course_folder("../docs", clear_existing=False).
  • Custom DevStaticFiles adds no-cache headers so frontend edits show immediately during development.
  • Mounts ../frontend at / as the static root.

rag_system.py — backend/rag_system.py

  • Doesn't do work itself — wires everyone else together.
  • Constructor builds: DocumentProcessor, VectorStore, AIGenerator, SessionManager, ToolManager, and registers CourseSearchTool.
  • add_course_document(path) → processes one file, adds metadata + chunks to the store.
  • add_course_folder(path, clear_existing=False) → batch ingests a directory; skips courses whose title is already in the catalog.
  • query(query, session_id?):
    • Builds prompt: "Answer this question about course materials: {query}"
    • Pulls conversation history from session manager (if session exists)
    • Calls ai_generator.generate_response(...) with tools
    • Reads tool_manager.get_last_sources(), resets them
    • Appends the exchange to session history
    • Returns (answer, sources)
  • get_course_analytics() → returns {total_courses, course_titles} for the sidebar.

document_processor.py — backend/document_processor.py

Expects course files in this format:

Course Title: Intro to MCP
Course Link: https://...
Course Instructor: Jane Doe

Lesson 1: Getting Started
Lesson Link: https://...
[lesson content...]

Lesson 2: Advanced Topics
[lesson content...]
  • read_file(path) — UTF-8 read with fallback to errors="ignore".
  • chunk_text(text) — sentence-aware splitter using a regex that handles abbreviations; greedy packs sentences up to chunk_size; carries back enough sentences to fill chunk_overlap.
  • process_course_document(path) — parses the 3-line header, walks lines detecting Lesson N: … markers and optional Lesson Link: lines, produces a Course + list of CourseChunk.
  • First chunk of each lesson is prefixed: "Lesson {N} content: ..." so the chunk carries its context into embeddings.

vector_store.py — backend/vector_store.py

Wraps ChromaDB with two collections:

CollectionHoldsUsed for
course_catalogCourse titles + JSON-serialized lesson metadataResolving partial course names ("MCP" → full title)
course_contentText chunks with metadata (course_title, lesson_number, chunk_index)The actual semantic search
  • search(query, course_name?, lesson_number?, limit?) — 3 steps: resolve course name via catalog vector search, build Chroma filter, run content query.
  • _resolve_course_name(name) — queries the catalog for top-1 match; returns the resolved title.
  • _build_filter(...) — composes $and filter for combined course+lesson, or single-field filter otherwise.
  • add_course_metadata(course) — uses the title as the Chroma ID and stores lessons as JSON in metadata.
  • add_course_content(chunks) — IDs are {title_with_underscores}_{chunk_index}.
  • Helpers: clear_all_data, get_existing_course_titles, get_course_count, get_all_courses_metadata, get_course_link, get_lesson_link.
  • Embedding function: SentenceTransformerEmbeddingFunction(model_name="all-MiniLM-L6-v2").

ai_generator.py — backend/ai_generator.py

  • Static SYSTEM_PROMPT constrains Claude: "One search per query maximum", no meta-commentary, brief educational answers.
  • Base API params: temperature=0, max_tokens=800.
  • generate_response(query, conversation_history?, tools?, tool_manager?):
    • Builds system by appending history if present
    • If tools provided, passes tool_choice={"type": "auto"}
    • If response has stop_reason == "tool_use", calls _handle_tool_execution
  • _handle_tool_execution(...) — appends the assistant tool-use turn, runs each tool, appends results as a user message containing tool_result blocks, then re-calls the API without tools to get the final text answer.

search_tools.py — backend/search_tools.py

  • Abstract Tool class with get_tool_definition() + execute(**kwargs).
  • CourseSearchTool exposes this Claude tool: 
    search_course_content(
   query:         "what to look for",
   course_name:   "optional — which course (partial match OK)",
   lesson_number: "optional — which lesson"
)
  • _format_results(...) formats each hit as [Course - Lesson N]\n{chunk} and stores human-readable sources in self.last_sources for the UI.
  • ToolManager registers tools, exposes get_tool_definitions(), execute_tool(name, **kwargs), and get_last_sources() / reset_sources().

session_manager.py — backend/session_manager.py

  • Pure in-memory store: Dict[str, List[Message]], lost on restart.
  • create_session() returns IDs like session_1, session_2, …
  • Trims to the last MAX_HISTORY * 2 = 4 messages per session.
  • get_conversation_history(session_id) returns a formatted transcript injected into the system prompt.

models.py — backend/models.py

  • Lesson — lesson_number: int, title: str, lesson_link: Optional[str].
  • Course — title (also used as unique ID), course_link, instructor, lessons: List[Lesson].
  • CourseChunk — content, course_title, lesson_number, chunk_index.

Key concepts — beginner cheat sheet

RAG (Retrieval-Augmented Generation)
Look up real info from a database first → then ask the AI to answer using that info. Much more accurate than asking the AI from training memory alone.
Embedding
Turning text into a list of numbers so computers can compare meaning. "puppy" and "young dog" get similar numbers — that's how semantic search works.
Vector database
A database that finds things by similarity of meaning, not by exact keyword match. This app uses ChromaDB.
Chunk
A small piece of a document — about 800 characters here. Documents get chopped into chunks so search can return specific relevant passages instead of whole files.
Tool use
Letting the AI call functions in your code. Claude decides when to call search_course_content() instead of you forcing a search every time.
Session
One ongoing conversation between user and bot. Sessions let the bot handle follow-ups like "tell me more about that."
System prompt
Hidden instructions to the AI sent on every call — sets the personality and rules (e.g. "be brief", "only search once per question").

Notable details, gotchas & opinionated choices

✅ Smart: Tool use means Claude decides whether to search. General-knowledge questions skip the database lookup entirely — faster and cheaper.
✅ Smart: Course titles are unique IDs in both Chroma collections. add_course_folder compares titles and skips re-ingesting courses already in the store.
✅ Smart: First chunk of each lesson is prefixed with "Lesson N content: ..." so embeddings carry lesson context — improves retrieval for lesson-specific questions.
⚠️ Limit: MAX_HISTORY = 2 means only the last 2 exchanges (4 messages) survive. Long conversations lose early context.
⚠️ Limit: The system prompt restricts Claude to one search per question. Multi-hop questions ("compare lesson 3 of course A with lesson 5 of course B") won't chain searches.
⚠️ Limit: Sessions live in process memory — restart the server and all chat history is lost.
🐛 Bug: get_lesson_link in vector_store.py:267 is missing an explicit return None on the exception path. It falls through to implicit None so it technically works, but reads as a bug.
⚠️ Limit: CORS and trusted hosts are wide open (allow_origins=["*"], allowed_hosts=["*"]) — fine for local dev, not safe for production.
💡 Tidbit: The chunker's first chunk uses "Lesson N content: ..." while the fallback path (last lesson processed) uses "Course {title} Lesson {N} content: ..." — slight inconsistency in chunk prefixing.

How to run it

# 1. Install dependencies (uses uv)
uv sync

# 2. Add your Anthropic API key to .env
ANTHROPIC_API_KEY=your_key_here

# 3. Launch
./run.sh
#   or manually:
cd backend && uv run uvicorn app:app --reload --port 8000

# 4. Open in browser
http://localhost:8000           # Chat UI
http://localhost:8000/docs      # Swagger API docs
